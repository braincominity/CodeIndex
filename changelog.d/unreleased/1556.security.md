## English

- **NuGet transitive dependencies (including native `SQLitePCLRaw.bundle_e_sqlite3`) are now pinned via committed `packages.lock.json`; CI restores in `--locked-mode` (#1556)** — `Directory.Build.props` enables `RestorePackagesWithLockFile=true` for every project in the solution (`src/CodeIndex`, `tests/CodeIndex.Tests`, and `tools/CodeIndex.Changelog`), and the freshly generated `packages.lock.json` files are committed alongside it. `dotnet.yml`, `release.yml`, and `codeql.yml` now pass `--locked-mode` to the solution-level `dotnet restore`, so a silent transitive bump or downgrade attack — including a native `e_sqlite3` swap — fails CI with `NU1004` instead of slipping into release artifacts. The release `dotnet publish` and `dotnet pack` steps deliberately do not enable `RestoreLockedMode` because their runtime-specific restores legitimately add `net8.0/<rid>` and `Microsoft.NET.ILLink.Tasks` entries; the locked versions are still consumed via `RestorePackagesWithLockFile=true`. Local development restores normally; the regeneration workflow (`dotnet restore --force-evaluate`, then commit the diff) is documented in `DEVELOPER_GUIDE.md`. NuGet cache keys in both workflows additionally hash `packages.lock.json` so a lock-file change correctly invalidates the cache.

## 日本語

- **NuGet の推移依存（ネイティブ実体を持つ `SQLitePCLRaw.bundle_e_sqlite3` を含む）を `packages.lock.json` の commit で固定し、CI を `--locked-mode` で restore する運用に切り替えました (#1556)** — `Directory.Build.props` でソリューション配下の全プロジェクト（`src/CodeIndex`, `tests/CodeIndex.Tests`, `tools/CodeIndex.Changelog`）に対し `RestorePackagesWithLockFile=true` を有効化し、生成した `packages.lock.json` を併せて commit します。`dotnet.yml` / `release.yml` / `codeql.yml` のソリューションレベル `dotnet restore` に `--locked-mode` を付与し、推移依存の暗黙バンプや downgrade attack（ネイティブ `e_sqlite3` の差し替えを含む）は release アーティファクトに紛れ込む前に `NU1004` で CI が失敗します。release の `dotnet publish` と `dotnet pack` には意図的に `RestoreLockedMode` を有効化していません。これらは `net8.0/<rid>` の runtime section や trimming 用 `Microsoft.NET.ILLink.Tasks` を正当に追加する runtime-specific restore を伴いますが、`RestorePackagesWithLockFile=true` により lock ファイル経由で固定版を解決します。ローカル開発では従来どおり `--locked-mode` なしで restore でき、依存更新時の再生成手順（`dotnet restore --force-evaluate` 実行→差分を同 PR に commit）は `DEVELOPER_GUIDE.md` に記載しました。両 workflow の NuGet キャッシュキーは `packages.lock.json` のハッシュも含むようにし、lock ファイル変更でキャッシュが正しく無効化されるようにしています。
