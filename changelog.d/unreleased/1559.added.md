## English

- **MCP server authentication framework with optional `CDIDX_MCP_AUTH_TOKEN` (#1559)** — Adds an `IMcpAuthenticator` middleware invoked once per JSON-RPC request before method dispatch, producing an `McpCallerIdentity` (source + subject) the future audit log (#1562) and any networked transports can consume. The default `LocalStdioAuthenticator` is permissive and preserves the historical stdio behaviour, so existing clients (Claude Code, Cursor, Windsurf) keep working without changes. Setting the `CDIDX_MCP_AUTH_TOKEN` environment variable to a non-whitespace value swaps in `TokenMcpAuthenticator`, which enforces a constant-time (`CryptographicOperations.FixedTimeEquals`) match against `params.auth.token` on every responded request — `initialize`, `tools/list`, `tools/call`, and `ping` — and rejects mismatches with a uniform JSON-RPC `-32001 "Unauthorized"` response so the wire body cannot distinguish missing-from-wrong tokens. Notifications (`notifications/initialized`, `notifications/cancelled`) still short-circuit before the auth gate because they produce no response and cannot signal an error code. The full failure reason is written to stderr via the new `BuildAuthFailureLog` helper for local diagnostics; the wire response stays sanitized per #1530.

## 日本語

- **MCP サーバーの認証フレームワークと任意設定の `CDIDX_MCP_AUTH_TOKEN` を追加しました (#1559)** — JSON-RPC リクエストごとに dispatch 前に 1 回呼ばれる `IMcpAuthenticator` ミドルウェアを追加し、`McpCallerIdentity` (source + subject) を生成して将来の監査ログ (#1562) やネットワーク transport から再利用できるようにしました。既定の `LocalStdioAuthenticator` は permissive で従来の stdio 動作を維持するため、既存クライアント (Claude Code、Cursor、Windsurf) は変更なしで動き続けます。環境変数 `CDIDX_MCP_AUTH_TOKEN` に空白以外の値を設定すると `TokenMcpAuthenticator` に切り替わり、`initialize` / `tools/list` / `tools/call` / `ping` のすべての応答対象リクエストで `params.auth.token` と定数時間比較 (`CryptographicOperations.FixedTimeEquals`) による一致を要求し、不一致は統一された JSON-RPC `-32001 "Unauthorized"` 応答で拒否します（ワイヤ本文では未提示と不一致を区別できないため、応答本文を通じたトークン存在判定オラクル攻撃を防止）。通知 (`notifications/initialized`、`notifications/cancelled`) は応答もエラーコードも持たないため、認証ゲートの前で従来通り short-circuit します。失敗詳細はローカル診断向けに `BuildAuthFailureLog` ヘルパーで stderr に出力され、ワイヤ応答は #1530 方針に従い sanitized のままです。
