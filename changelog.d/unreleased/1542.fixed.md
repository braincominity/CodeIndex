## English

- **Indexer no longer stalls on multi-MB single-line files (#1542)** — `ChunkSplitter`, `SymbolExtractor`, and `ReferenceExtractor` now share a 64 KB per-line cap (`ChunkSplitter.MaxLineLength`). When any single physical line exceeds the cap (e.g. a 1 MB minified `*.min.js` bundle or a base64-encoded asset), all three return empty for that file so a multi-MB Content column is not persisted and regex-based extraction cannot stall on catastrophic backtracking / OOM. The skip is surfaced as a `line_too_long` `FileIssue` from `FileIndexer.ValidateContent` so the previously silent stall is now observable through the existing issues channel.

## 日本語

- **複数 MB の単一行ファイルでインデクサが停止する問題を解消 (#1542)** — `ChunkSplitter`、`SymbolExtractor`、`ReferenceExtractor` が 64 KB の行ごとの上限 (`ChunkSplitter.MaxLineLength`) を共有するようになりました。任意の物理行が上限を超えるファイル (例: 1 MB の minified `*.min.js` バンドルや base64 エンコードされたアセット) では 3 者とも当該ファイルに対して空を返し、複数 MB の Content カラムが永続化されず、正規表現ベースの抽出が catastrophic backtracking / OOM で停止しないようにします。スキップは `FileIndexer.ValidateContent` から `line_too_long` の `FileIssue` として既存の issues 経路で観測できるため、以前は無音の停止だった症状が可視化されます。
