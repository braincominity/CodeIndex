## English

- **`impact` truncation now reports `truncated_reason` (#1533)** — when `cdidx impact` (and the `impact_analysis` MCP tool) returns `truncated: true`, the JSON payload now also emits a `truncated_reason` field with one of two values: `user_limit` (the user-supplied `--limit` was reached — raising `--limit` will return more results) or `safety_cap` (an internal per-symbol BFS fetch-iteration cap fired — the input graph is likely pathological and raising `--limit` alone will not help). The MCP summary tail and CLI text-mode footer also surface the reason, giving AI/MCP consumers actionable retry guidance instead of guessing why truncation happened. The field is omitted when `truncated: false`.

## 日本語

- **`impact` の truncation 通知に `truncated_reason` を追加 (#1533)** — `cdidx impact`（および `impact_analysis` MCP ツール）が `truncated: true` を返すとき、JSON ペイロードに `truncated_reason` フィールドが追加されました。値は次の 2 種類のいずれかです: `user_limit`（ユーザ指定の `--limit` に到達したことを示し、`--limit` を上げれば追加結果を取得可能）または `safety_cap`（シンボル単位の BFS フェッチ反復セーフティ上限に達したことを示し、グラフが病的なため `--limit` を上げても解消しない）。MCP のサマリ末尾および CLI テキストモードのフッタにも理由が表示されるため、AI/MCP クライアントは「なぜ打ち切られたのか」を推測することなく適切な再試行ガイダンスを取得できます。`truncated: false` の場合はフィールド自体が省略されます。
