## English

- **`DbReader` reuses a connection-scoped schema cache instead of re-running `PRAGMA table_info` / `PRAGMA index_list` per construction (#1565)** — the new `DbSchemaCache` (attached to `DbContext.SchemaCache`) memoizes the `files` / `symbols` / `symbol_references` column sets, the `symbols` / `symbol_references` index sets, and `sqlite_master` table-existence checks. The MCP `WithDbReader` hot path and the CLI `WithDb` reader path now route through the cache, so each subsequent tool call pays the schema scan once per session instead of once per `DbReader`. `DbContext.RefreshSchemaCache()` is exposed for callers that mutate the schema out of band; `InitializeSchema`, `TryMigrateForRead`, and `DropAll` invalidate the cache automatically. The legacy `(SqliteConnection, bool)` constructor still falls back to inline PRAGMA queries so write-side signal readers that hand in a raw connection continue to work unchanged.

## 日本語

- **`DbReader` がコネクション単位のスキーマキャッシュを再利用するようになり、construction ごとの `PRAGMA table_info` / `PRAGMA index_list` 再走査を排除 (#1565)** — 新規追加した `DbSchemaCache` (`DbContext.SchemaCache` から取得) が `files` / `symbols` / `symbol_references` の列集合・`symbols` / `symbol_references` の index 集合・`sqlite_master` のテーブル存在判定を memoize します。MCP の `WithDbReader` ホットパスおよび CLI の `WithDb` リーダ生成経路はこのキャッシュ経由となり、ツール呼び出しごとではなくセッションごとに 1 回だけスキーマ走査を行います。スキーマを外部から変更したケースに備え `DbContext.RefreshSchemaCache()` を公開しており、`InitializeSchema` / `TryMigrateForRead` / `DropAll` は自動でキャッシュを破棄します。従来の `(SqliteConnection, bool)` コンストラクタは引き続き利用可能で、その経路ではキャッシュを使わずインラインで PRAGMA を発行するため、書き込み側でシグナル取得用にコネクションだけ渡している呼び出し元の動作は維持されます。
