## English

- **Windows long-path support: deep monorepo paths are no longer silently skipped (#1547)** — `FileIndexer` previously called `Directory.EnumerateFiles` / `Directory.EnumerateDirectories` / `File.GetAttributes` / `File.OpenRead` / `new FileStream(...)` / `File.GetLastWriteTimeUtc` / `File.ReadAllLines` with bare absolute paths, and the `cdidx.exe` apphost did not declare itself long-path-aware. On Windows that combination caused any path crossing `MAX_PATH` (260) — typical in `node_modules/.pnpm/<pkg>@<ver>/node_modules/...` trees — to surface as `PathTooLongException` / `DirectoryNotFoundException` and be silently dropped from the index. The fix has three layers that work together: (1) the `cdidx` project now embeds a Windows app manifest with `<longPathAware>true</longPathAware>` so the published apphost honors the OS-level `LongPathsEnabled` registry opt-in; (2) a new `CodeIndex.Indexer.LongPath` helper wraps absolute Windows paths >= 248 characters with the `\\?\` (drive) or `\\?\UNC\` (UNC) extended-length prefix and is a no-op for short / relative / device-namespace / non-Windows / already-prefixed inputs; (3) every walker I/O syscall in `FileIndexer` (`Directory.Enumerate*`, `File.Exists` / `File.GetAttributes` / `File.OpenRead` / `File.ReadLines` / `File.ReadAllLines` / `File.WriteAllText` / `File.Delete` / `new FileStream(...)` / `File.GetLastWriteTimeUtc`, including the `.gitmodules` reader in `LoadGitSubmodulePaths`) plus the downstream `File.Exists` checks that decide which DB rows to keep on every index run (`DbWriter.PurgeStaleFiles`, `IndexCommandRunner` `--update-paths` / `--commits` / per-file update loop) and the `DbPathResolver` checksum sampling are routed through `LongPath.EnsureWindowsPrefix(...)`; entries returned from `Enumerate*` are stripped via `LongPath.RemoveWindowsPrefix(...)` before any `Path.GetRelativePath(_projectRoot, …)` / `NormalizeScopeKey(...)` so canonical project-relative keys stay identical to the short-path case. Without the layer (3) `PurgeStaleFiles` patch, a deep file indexed successfully would have been classified as missing and DELETED from the DB on the next index run, undoing the layer (2) walker fix. POSIX behavior is unchanged because `EnsureWindowsPrefix` short-circuits when `RuntimeInformation.IsOSPlatform(OSPlatform.Windows)` is false; cross-platform unit tests target the internal `EnsureWindowsPrefixCore` / `RemoveWindowsPrefixCore` overloads with an explicit `isWindows` flag so the prefix logic is verified on POSIX CI.

## 日本語

- **Windows ロングパス対応: 深い monorepo パスが silent skip されないようになりました (#1547)** — `FileIndexer` は従来 `Directory.EnumerateFiles` / `Directory.EnumerateDirectories` / `File.GetAttributes` / `File.OpenRead` / `new FileStream(...)` / `File.GetLastWriteTimeUtc` / `File.ReadAllLines` を絶対パスのまま呼んでおり、`cdidx.exe` apphost も long-path-aware として宣言されていませんでした。Windows ではこの組み合わせにより `MAX_PATH` (260) を超えるパス（典型的には `node_modules/.pnpm/<pkg>@<ver>/node_modules/...` 配下）が `PathTooLongException` / `DirectoryNotFoundException` として落ち、インデックスから黙って漏れていました。修正は 3 層構成です: (1) `cdidx` プロジェクトに `<longPathAware>true</longPathAware>` を含む Windows app manifest を埋め込み、公開 apphost が OS 側 `LongPathsEnabled` レジストリ opt-in を尊重するようにします。(2) 新規 `CodeIndex.Indexer.LongPath` ヘルパーが Windows 絶対パス（>= 248 文字）に `\\?\`（ドライブ）/ `\\?\UNC\`（UNC）拡張長接頭辞を付与し、短い・相対・デバイス名前空間・非 Windows・既に prefix 済みの入力には no-op です。(3) `FileIndexer` の walker I/O syscall（`Directory.Enumerate*`、`File.Exists` / `File.GetAttributes` / `File.OpenRead` / `File.ReadLines` / `File.ReadAllLines` / `File.WriteAllText` / `File.Delete` / `new FileStream(...)` / `File.GetLastWriteTimeUtc`、`LoadGitSubmodulePaths` の `.gitmodules` 読み取りを含む）に加え、毎回の index 実行で DB 行を残すか判定する下流の `File.Exists` 各所（`DbWriter.PurgeStaleFiles`、`IndexCommandRunner` の `--update-paths` / `--commits` / per-file update ループ）と `DbPathResolver` の checksum サンプリングを `LongPath.EnsureWindowsPrefix(...)` 経由に統一しました。`Enumerate*` から返ったエントリは `LongPath.RemoveWindowsPrefix(...)` で剥がしてから `Path.GetRelativePath(_projectRoot, …)` / `NormalizeScopeKey(...)` に渡すため、canonical な project-relative key は短パス時と同一に保たれます。(3) の `PurgeStaleFiles` を直さないと、(2) で正しく index した深いファイルが次回の index 実行で「存在しない」と判定されて DB から DELETE され、walker の修正が無に帰す回帰になっていました。POSIX 側の振る舞いは変わりません — `EnsureWindowsPrefix` は `RuntimeInformation.IsOSPlatform(OSPlatform.Windows)` が false なら short-circuit します。クロスプラットフォーム単体テストは `isWindows` を明示的に取る内部 `EnsureWindowsPrefixCore` / `RemoveWindowsPrefixCore` オーバーロードを使うことで、POSIX CI でも prefix ロジックを検証できます。
