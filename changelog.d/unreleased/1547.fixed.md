## English

- **Windows long-path support: deep monorepo paths are no longer silently skipped (#1547)** — `FileIndexer` previously called `Directory.EnumerateFiles` / `Directory.EnumerateDirectories` / `File.GetAttributes` / `File.OpenRead` / `new FileStream(...)` / `File.GetLastWriteTimeUtc` with bare absolute paths, and the `cdidx.exe` apphost did not declare itself long-path-aware. On Windows that combination caused any path crossing `MAX_PATH` (260) — typical in `node_modules/.pnpm/<pkg>@<ver>/node_modules/...` trees — to surface as `PathTooLongException` / `DirectoryNotFoundException` and be silently dropped from the index. The fix has two layers that work together: (1) the `cdidx` project now embeds a Windows app manifest with `<longPathAware>true</longPathAware>` so the published apphost honors the OS-level `LongPathsEnabled` registry opt-in; (2) a new `CodeIndex.Indexer.LongPath` helper wraps every walker I/O syscall with `LongPath.EnsureWindowsPrefix(...)` (`\\?\` for drive-absolute paths, `\\?\UNC\` for UNC paths, no-op for short / relative / device-namespace / non-Windows / already-prefixed inputs) at a 248-character threshold, and entries that come back from `Enumerate*` are stripped via `LongPath.RemoveWindowsPrefix(...)` before any `Path.GetRelativePath(_projectRoot, …)` / `NormalizeScopeKey(...)` so canonical relative paths and scope keys stay identical to the short-path case. POSIX behavior is unchanged because `EnsureWindowsPrefix` short-circuits when `RuntimeInformation.IsOSPlatform(OSPlatform.Windows)` is false; cross-platform unit tests target the internal `EnsureWindowsPrefixCore` / `RemoveWindowsPrefixCore` overloads with an explicit `isWindows` flag so the prefix logic is verified on POSIX CI.

## 日本語

- **Windows ロングパス対応: 深い monorepo パスが silent skip されないようになりました (#1547)** — `FileIndexer` は従来 `Directory.EnumerateFiles` / `Directory.EnumerateDirectories` / `File.GetAttributes` / `File.OpenRead` / `new FileStream(...)` / `File.GetLastWriteTimeUtc` を絶対パスのまま呼んでおり、`cdidx.exe` apphost も long-path-aware として宣言されていませんでした。Windows ではこの組み合わせにより `MAX_PATH` (260) を超えるパス（典型的には `node_modules/.pnpm/<pkg>@<ver>/node_modules/...` 配下）が `PathTooLongException` / `DirectoryNotFoundException` として落ち、インデックスから黙って漏れていました。修正は 2 層構成です: (1) `cdidx` プロジェクトに `<longPathAware>true</longPathAware>` を含む Windows app manifest を埋め込み、公開 apphost が OS 側 `LongPathsEnabled` レジストリ opt-in を尊重するようにします。(2) 新規 `CodeIndex.Indexer.LongPath` ヘルパーが walker のすべての I/O syscall を `LongPath.EnsureWindowsPrefix(...)`（ドライブ絶対パスは `\\?\`、UNC は `\\?\UNC\`、短い・相対・デバイス名前空間・非 Windows・既に prefix 済みは no-op）で 248 文字閾値で wrap し、`Enumerate*` から返ったエントリは `LongPath.RemoveWindowsPrefix(...)` で剥がしてから `Path.GetRelativePath(_projectRoot, …)` / `NormalizeScopeKey(...)` に渡すため、canonical な相対パスや scope key は短パス時と同一に保たれます。POSIX 側の振る舞いは変わりません — `EnsureWindowsPrefix` は `RuntimeInformation.IsOSPlatform(OSPlatform.Windows)` が false なら short-circuit します。クロスプラットフォーム単体テストは `isWindows` を明示的に取る内部 `EnsureWindowsPrefixCore` / `RemoveWindowsPrefixCore` オーバーロードを使うことで、POSIX CI でも prefix ロジックを検証できます。
