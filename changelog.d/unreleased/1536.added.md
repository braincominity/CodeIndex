## English

- **`impact` can now surface the intermediate symbols on each transitive caller chain via opt-in `--with-paths` (#1536)** — Previously `ImpactResult` exposed only `Depth`, so when BFS converged on the same outer caller through distinct intermediates (e.g. `A → B → foo` and `A → C → foo`) the dedup collapsed those routes into a single row and the "via what" information was lost. Pass `cdidx impact <symbol> --with-paths` (CLI) or `withPaths: true` (MCP `impact_analysis`) to populate a new optional `paths` field on each caller — a list of shortest call chains ordered `[resolvedRoot, intermediate..., callerName]`, with every same-depth convergence surfaced as a distinct entry. JSON omits the field by default to keep payloads compact; a per-row cap (10 paths) is applied with a `paths_truncated` boolean when there are more shortest routes than emitted. Default behavior (without the flag) is unchanged.

## 日本語

- **`impact` の transitive caller chain について、`--with-paths` で中間シンボルを含む経路を出力できるようにしました (#1536)** — 従来の `ImpactResult` は `Depth` しか持たず、BFS が同名 caller に異なる中間経由 (`A → B → foo` と `A → C → foo`) で収束する場合、dedup により 1 行にまとめられて「どの経由で繋がるか」が失われていました。`cdidx impact <symbol> --with-paths`（CLI）または `withPaths: true`（MCP `impact_analysis`）を指定すると、各 caller に `paths` フィールドが付き、`[resolvedRoot, 中間..., callerName]` の順で最短経路を列挙します。同 depth で複数経路が収束する場合（ダイヤモンド型）はすべての経路が別エントリとして出力されます。既定の JSON は本フィールドを省略して payload を小さく保ち、1 行あたりの保持上限（10 経路）を超える経路が存在する場合は `paths_truncated` を `true` にして通知します。フラグ未指定時の既存挙動は変更ありません。
