## English

- **`validate` flags bare CR (legacy Mac) and three-way line-ending mixes (#1538)** — `FileIndexer.ValidateContent` previously only emitted `mixed_line_endings` for files that contained both CRLF and LF, and `BuildRecordWithRawBytes`'s `"\r\n" → "\n"` then `"\r" → "\n"` normalization silently flattened bare CR (legacy Mac) endings, leaving line-counting / regex assumptions unverified for SVN-imported sources. Validation now scans raw bytes for CR, CRLF, and LF independently and emits a new `cr_only_line_endings` issue when only bare CR is present and a `mixed_line_endings_three_way` issue when all three coexist; two-way CR + (CRLF | LF) mixes still surface as `mixed_line_endings` with a descriptive message. The new kinds appear in `cdidx validate` / `--kind` filters and the MCP `validate` tool description.

## 日本語

- **`validate` が CR-only (旧 Mac) と 3 種混在の改行を検出するようになりました (#1538)** — `FileIndexer.ValidateContent` は従来 CRLF と LF が両方ある場合のみ `mixed_line_endings` を出し、`BuildRecordWithRawBytes` の `"\r\n" → "\n"` → `"\r" → "\n"` 正規化が CR-only (旧 Mac) 改行を黙って潰していたため、SVN 由来などの古いファイルで行カウント・正規表現の前提崩れが見えませんでした。検証段で raw bytes を 1 パス走査し、CR / CRLF / LF を独立に判定するように変更。CR のみなら `cr_only_line_endings`、3 種すべてが混在すれば `mixed_line_endings_three_way` の `FileIssue` を新設し、CR + (CRLF または LF) の 2 種混在も既存 `mixed_line_endings` に説明付きメッセージで集約します。新 kind は `cdidx validate` / `--kind` フィルタおよび MCP `validate` ツール記述からも参照できます。
