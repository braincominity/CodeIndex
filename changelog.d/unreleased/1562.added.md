## English

- **Opt-in structured audit log for MCP tool invocations (#1562)** — `cdidx mcp` now accepts `--audit-log <path>` (with companion `--audit-log-include-values` and `--audit-log-max-bytes <n>` flags) to emit one JSONL record per `tools/call`. Each record captures the timestamp, MCP tool name, caller (`initialize.clientInfo.{name,version}`), request id, argument *keys* and per-key length sketch, observed `result_count`, wall-clock `elapsed_ms`, and an `error_code` that maps to `0` on success, `1` for tool errors, or the verbatim JSON-RPC error code for protocol failures. Argument *values* are off by default and gated behind `--audit-log-include-values` because cdidx queries can carry source snippets or secret-shaped strings. Malformed `tools/call` requests (missing tool name), unknown tools, and operator-disabled tools (#1561 deny-list) are all audited so a misbehaving client cannot hide its activity by varying the request shape and so deny-list policy is auditable after the fact. The destination is probe-opened at construction so misconfigured paths (existing directory, denied parent) fail loudly before the server claims auditing is on; misconfigured `--db --audit-log` argument ordering preserves the `--db` value instead of consuming it as an audit flag; and every fresh `initialize` reseats caller identity so a reconnect that omits `clientInfo` cannot inherit the previous client's name on subsequent records. The log file is opened append-only, rotated through `<path>.1` / `<path>.2` at the configured size cap (default 50 MiB, minimum 4 KiB), and capped at three slots total so a slow drain of the log cannot fill the disk. Emission is best-effort once construction succeeds: any later IO failure is swallowed so audit writing never breaks the underlying tool call. Compliance reviewers can now answer "who called which tool with what shape of arguments and when did it fail?" without re-running the index.

## 日本語

- **MCP ツール呼び出しのオプトイン監査ログを追加 (#1562)** — `cdidx mcp` に `--audit-log <path>`（および `--audit-log-include-values`、`--audit-log-max-bytes <n>`）を追加し、`tools/call` ごとに 1 行の JSONL レコードを出力できるようになりました。各レコードには開始時刻、MCP ツール名、呼び出し元（`initialize.clientInfo.{name,version}`）、リクエスト id、引数の *キー* と長さ概算、`result_count`、`elapsed_ms`、`error_code`（成功=`0`、ツールエラー=`1`、JSON-RPC エラー=そのコード）を記録します。引数の *値* は既定で出力せず、`--audit-log-include-values` 指定時のみ含めます（cdidx クエリにはソース片や secret 風文字列が混入しうるため）。引数欠落の `tools/call`、未知ツール、オペレータが無効化したツール（#1561 deny-list）もすべて監査対象にし、リクエスト形状を変えて監査から消える挙動を防ぐとともに deny-list ポリシーの効果を後追いできるようにします。構築時に append open を probe して、不正な出力先（既存ディレクトリや書き込み不可な親）は audit 有効と表示する前に失敗させます。`--db --audit-log` のように `--db` の値がダッシュ始まりで audit フラグと紛らわしいケースでも `--db` の値はそのまま保たれます。再 initialize が `clientInfo` を省略した場合は前回クライアント名を引き継がず caller フィールドをクリアします。出力先は append 専用で開かれ、サイズ上限（既定 50 MiB、最小 4 KiB）を超えると `<path>.1` → `<path>.2` の順にローテーションされ、最古スロットは破棄されます（合計 3 スロットに上限）。構築後の IO 失敗は best-effort で握り潰して本体ツール呼び出しを壊しません。コンプライアンス監査で「誰が・どんな引数形で・いつ呼び出して失敗したか」を index を再実行せずに後追いできます。
