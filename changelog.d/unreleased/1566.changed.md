## English

- **Reuse prepared statements on hot per-file write paths (#1566)** — `DbWriter` now leases per-file `SqliteCommand` instances from a bounded LRU `PreparedCommandCache` shared with its owning `DbContext`, so per-file lookups (`GetUnchangedFileId`, `HasFileAtPath`, `CleanExistingFileData`), the `UpsertFile` INSERT…ON CONFLICT, the four `DeleteFileData` deletes, and the `DeleteFileByPath` delete all hit one parsed/planned statement per SQL text instead of constructing and disposing a fresh command per call. Indexing a 10k-file project previously paid ~100k extra parse/plan cycles on these hot paths; with this change each statement is parsed once per session and rebinds parameter values for subsequent calls. Cached commands are re-pointed at the writer's active `SqliteTransaction` on every lease so reuse remains safe across nested savepoints and transaction boundaries. The cache is opt-in via the new `DbWriter(DbContext)` constructor — the legacy `DbWriter(SqliteConnection)` ctor still works without caching, preserving every existing test and external call site.

## 日本語

- **ファイル単位のホット書き込みパスで prepared statement を再利用 (#1566)** — `DbWriter` がファイル単位の `SqliteCommand` を所属 `DbContext` と共有するサイズ制限付き LRU `PreparedCommandCache` から借りるようになりました。これにより `GetUnchangedFileId` / `HasFileAtPath` / `CleanExistingFileData` の lookup、`UpsertFile` の INSERT…ON CONFLICT、`DeleteFileData` の 4 本の DELETE、`DeleteFileByPath` の DELETE が、いずれも SQL テキストごとに 1 つの parse/plan 済み statement を再利用する形に変わりました。10k ファイル規模の index 実行では、これらホットパスだけで以前は ~100k 回の parse/plan が走っていましたが、今後はセッションあたり 1 回 prepare して以降はパラメータ値を rebind するだけになります。キャッシュ済み command は借り出し時に writer の現在の `SqliteTransaction` に再バインドされるため、ネストされた savepoint やトランザクション境界をまたいでも再利用が安全です。キャッシュは新しい `DbWriter(DbContext)` コンストラクタからの opt-in であり、既存の `DbWriter(SqliteConnection)` コンストラクタはキャッシュなしで従来通り動作するため、テストや外部呼び出し側に手を入れる必要はありません。
