## English

- **Path case-sensitivity now probes the actual filesystem rather than the host OS family (#1546)** — `PathsEqual` / `IsPathEqualOrParent` previously keyed comparisons on `OperatingSystem.IsWindows()`, so case-sensitive APFS volumes on macOS, case-sensitive WSL NTFS / dev-drives, and case-sensitive ReFS mounts were misclassified (collapsing distinct files like `Foo.cs` vs `foo.cs`, or failing to match equivalent paths on a Windows case-sensitive directory). All four duplicated comparison sites (`DbPathResolver`, `FileIndexer`, `IndexCommandRunner`, `McpToolHandlers`) now delegate to a single `PathCasing` helper that probes by creating a case-variant of an existing ancestor directory, caches the result per anchor, and seeds itself from the workspace `core.ignorecase` + probe at index time. `cdidx status` (human and `--json`) now exposes the resolved case-sensitivity as `path_case_sensitive` so this trust decision is auditable from outside the indexer.

## 日本語

- **パスの大小区別判定が、ホスト OS の系列ではなく実ファイルシステムをプローブして決まるようになりました (#1546)** — これまで `PathsEqual` / `IsPathEqualOrParent` は `OperatingSystem.IsWindows()` だけで比較モードを切り替えていたため、macOS 上の case-sensitive APFS ボリューム、WSL の case-sensitive NTFS / dev-drive、case-sensitive ReFS マウントが誤分類され、`Foo.cs` と `foo.cs` が同一視されたり、Windows 上の case-sensitive ディレクトリで同一パスとして突き合わせられない状況が発生していました。重複していた 4 箇所の比較ロジック (`DbPathResolver` / `FileIndexer` / `IndexCommandRunner` / `McpToolHandlers`) はすべて `PathCasing` ヘルパー 1 つに集約され、最寄りの既存ディレクトリに case-variant を当てて存在判定し、結果をアンカー単位でキャッシュします。インデックス時には `core.ignorecase` + プローブ結果でキャッシュをシードし、`cdidx status`（human / `--json` の両方）に `path_case_sensitive` として公開されるため、インデクサ外部からもこの信頼判定を監査できます。
