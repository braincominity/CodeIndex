## English

- **`impact_analysis` `maxDepth` cap raised to 50 and clamping is now surfaced instead of silent (#1534)** — the MCP `impact_analysis` tool previously clamped `maxDepth` to `[0, 10]` without telling the caller, so a request for `maxDepth: 100` was silently downgraded to `10` and deep monorepo caller chains (DI container → factory → service → handler → business logic, ≥5 hops) underreported blast radius. The server-side cap is now `50`, and when the user-supplied value exceeds it the response payload adds a `warnings` array (`"maxDepth was clamped from 100 to 50 (server cap is [0, 50])."`) plus a `max_depth_requested` field carrying the original request, and the human-readable `content` text appends the same warning. The cap is documented in the MCP tool schema (`maximum: 50`, plus a description note about the `warnings` / `max_depth_requested` contract) so agents can see the bound before issuing a request. Default `maxDepth` and the BFS bound semantics are unchanged; only the upper cap and visibility behaviour change.
