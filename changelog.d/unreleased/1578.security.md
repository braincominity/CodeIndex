## English

- **`cdidx index --verbose` no longer leaks stack traces to stderr (#1578)** — Per-file error handling in `IndexCommandRunner` previously appended `ex.StackTrace` to stderr under `--verbose`, exposing internal type names, source paths, and line numbers to anything that captured the indexer's stderr (notably MCP clients that ingest child-process stderr for diagnostics). Both the full-scan and `--files` / `--commits` update-mode catch blocks now share a single `FormatPerFileErrorLine` helper that emits a one-line `  [ERR ] <path>: <message>` regardless of verbose state, with CR/LF in the path or exception message collapsed to a space so a multiline exception cannot inject pseudo-stack lines either. Public stderr stays a user-actionable message and stack frames are no longer reachable through the MCP transcript. Detailed crash context is still available via `cdidx report` and the opt-in `CDIDX_DEBUG` channel.

## 日本語

- **`cdidx index --verbose` が stderr にスタックトレースを漏らさないように修正しました (#1578)** — `IndexCommandRunner` のファイル単位エラー処理は `--verbose` 時に `ex.StackTrace` を stderr へ追記しており、内部型名・ソースパス・行番号が、インデクサの stderr を取り込むあらゆる消費者（特に診断目的で子プロセスの stderr を取り込む MCP クライアント）に漏れていました。フルスキャンと `--files` / `--commits` 更新モードの両方の catch ブロックで共通の `FormatPerFileErrorLine` ヘルパーを使い、verbose の有無に関係なく `  [ERR ] <path>: <message>` の 1 行のみを出力するようになり、さらに path や例外メッセージに含まれる CR/LF は空白へ畳み込まれるため、複数行の例外メッセージで疑似スタック行が注入されることもなくなりました。公開 stderr はユーザー向けメッセージだけとなり、MCP トランスクリプト経由でスタックフレームに到達できなくなります。詳細なクラッシュ情報は引き続き `cdidx report` および opt-in の `CDIDX_DEBUG` 経由で取得できます。
